# デプロイ前リファクタ指針（2026-06-25）

## このドキュメントの目的

デプロイ前レビューで抽出した「セキュリティ欠陥」と「性能劣化リスク」を、実装で解消するための具体方針をまとめる。  
本書は後続の実装作業で「なぜ直すか」「どう直すか」の判断基準として使う。

## 前提（今回の実装方針）

- 前回の認識合わせがセッション中断で取得できなかったため、以下の**推奨デフォルト**で進める。
- 追加インフラは最小化しつつ、複数インスタンスでも破綻しない設計を優先する。
- 認証方式は `SPEC.md` の方針（Bearer JWT）を維持する。

---

## 対応項目一覧（なぜ必要か / どう対応するか）

### 1) OpenAI エンドポイントの乱用対策不足

- **なぜ必要か**
  - 会話・フィードバック API は高コスト処理で、流量制御なしだと課金爆発と DoS リスクがある。
- **対応方針**
  - DB 永続のレート制限基盤を導入し、ユーザー単位で `minute/day` の二段制限を実装する。
  - しきい値は環境変数化し、運用で調整可能にする。

### 2) `X-Forwarded-For` 無検証利用

- **なぜ必要か**
  - 任意ヘッダーを信頼するとレート制限回避が可能になる。
- **対応方針**
  - `TRUST_X_FORWARDED_FOR` を導入し、既定は `false`。
  - `true` 時のみ XFF を使い、それ以外は `scope.client` を利用する。

### 3) register エンドポイントのレート制限不足

- **なぜ必要か**
  - アカウント乱造・スパム登録の温床になる。
- **対応方針**
  - 既存ログイン制限に加え、`/auth/register` にも同等のレート制限を適用する。

### 4) `scenario_text` 無制限

- **なぜ必要か**
  - 異常に長い入力で DB・OpenAI コスト・応答遅延が増加する。
- **対応方針**
  - `scenario_text` に上限を設定（既定 2000 文字、環境変数化）。

### 5) OpenAI クライアントの毎回生成

- **なぜ必要か**
  - コネクション再初期化で遅延・リソース消費が増える。
- **対応方針**
  - `AsyncOpenAI` を共有シングルトン化し、会話/添削サービスで共通利用する。

### 6) CORS 設定の過剰許可

- **なぜ必要か**
  - `allow_credentials=True` とワイルドカード許可は不要な攻撃面を増やす。
- **対応方針**
  - Bearer 前提に合わせ `allow_credentials=False`。
  - `allow_methods` / `allow_headers` を必要最小限に制限する。

### 7) メール検証未実装（仕様明確化不足）

- **なぜ必要か**
  - セキュリティ要求の期待値が曖昧だと運用事故につながる。
- **対応方針**
  - SMTP 基盤未導入のため、MVP では「メール検証なし」を明示し、今後の導入条件を docs に記載する。

### 8) フロント JWT localStorage 保存

- **なぜ必要か**
  - XSS 時にトークン奪取リスクがある。
- **対応方針**
  - Bearer 方針は維持しつつ、リスクと緩和策（有効期限短縮、CSP、入力サニタイズ方針）を docs に明記する。

### 9) DB 制約不足（`status/role/type`）

- **なぜ必要か**
  - アプリバグ時に不正値が混入し、後続処理の障害原因になる。
- **対応方針**
  - Alembic マイグレーションで `CHECK` 制約を追加する。

### 10) DB プール設定がデフォルト依存

- **なぜ必要か**
  - インスタンス増加時に Neon 接続上限を超過しやすい。
- **対応方針**
  - SQLAlchemy / asyncpg のプールサイズを環境変数化して明示設定する。

### 11) レート制限がプロセスローカル

- **なぜ必要か**
  - 複数インスタンスで制限が分散して実効性が落ちる。
- **対応方針**
  - DB 永続のレート制限テーブルを導入し、全インスタンス共有の判定にする。

### 12) レート制限メモリリーク

- **なぜ必要か**
  - 変動するキーで辞書が増え続ける。
- **対応方針**
  - インメモリ辞書方式を廃止し、DB 方式へ統一して解消する。

### 13) フィードバック対象メッセージ無制限

- **なぜ必要か**
  - 長会話で OpenAI 入力が肥大し、遅延と費用が悪化する。
- **対応方針**
  - フィードバック入力は「直近 N 件（既定 100）」に制限し、時系列は維持する。

### 14) 保存 API の N 回 POST

- **なぜ必要か**
  - 指摘数が多いと往復回数が増え遅くなる。
- **対応方針**
  - バルク保存 API（`POST /api/saved-items/batch`）を追加し、フロントは一括送信に切り替える。

### 15) `.dockerignore` 不在（backend）

- **なぜ必要か**
  - 不要ファイル混入でイメージ肥大・秘密情報混入リスクがある。
- **対応方針**
  - `backend/.dockerignore` を追加し、`.venv`, キャッシュ, `.env` 系を除外する。

### 16) backend コンテナ root 実行

- **なぜ必要か**
  - コンテナ侵害時の被害範囲が広がる。
- **対応方針**
  - Dockerfile を non-root 実行に変更する。

### 17) frontend Dockerfile が dev サーバー用途

- **なぜ必要か**
  - 本番で誤利用されるとセキュリティ/性能要件を満たしにくい。
- **対応方針**
  - 開発専用であることを明示し、`SETUP/SPEC` に本番は Cloudflare Pages 利用を追記する。
